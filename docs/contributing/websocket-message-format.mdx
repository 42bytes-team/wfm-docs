# WebSocket Message Format

Use this page as the Markdown/MDX reference for WebSocket documentation style. The examples are illustrative format samples, not public API contracts.

## Visual Language

WebSocket pages should stay mostly Markdown, with `WsMessage` cards for message routes and API UI components for badges, callouts, and field types.

<BadgeGrid
  badges={[
    {kind: 'neutral', icon: '💬', label: 'Command'},
    {kind: 'neutral', icon: '🔔', label: 'Event'},
    {kind: 'neutral', icon: '🔁', label: 'Stream'},
    {kind: 'auth', icon: '🔒', label: 'Auth required'},
    {kind: 'danger', icon: '🚧', label: 'Not implemented'},
  ]}
/>

### Conventions

- Use `WsMessage` for every documented route.
- Keep badges text-first; icons and emoji are visual aids, not the only meaning.
- Split command docs into `Client Message`, `Payload Fields`, `Success Message`, and `Error Message` sections.
- Split event docs into a message example and `Payload Fields` when the payload is inline.
- Use `json` for concrete copyable messages.
- Use `ts` for schematic messages that contain reusable model types, such as `payload: Order`.
- Link reusable models to [Data Models](../data-models.mdx) instead of duplicating full schemas.
- Use `ApiCallout` for auth requirements, restrictions, stream behavior, and implementation notes.
- Do not use raw `:::warning` blocks unless directive rendering has been verified.

## Message Type Table

Render message markers directly in tables when explaining message types.

| Marker | Meaning |
|---|---|
| <ApiBadge kind="neutral" icon="💬" label="Command" /> | Client request that expects `:ok`, `:error`, or protection error response. |
| <ApiBadge kind="neutral" icon="🔔" label="Event" /> | One-way server message that does not expect a response. |
| <ApiBadge kind="neutral" icon="🔁" label="Stream" /> | Event sequence with `meta.stream` and `meta.revision` for ordering. |
| <ApiBadge kind="auth" icon="🔒" label="Auth required" /> | Message requires an authenticated and authorized client. |
| <ApiBadge kind="danger" icon="🚧" label="Not implemented" /> | Route or behavior exists only as a planned/stubbed feature. |

## Command Example

<WsMessage
  type="Command"
  route="@wfm|cmd/example/doThing"
  summary="Perform an example action and return a command response."
  badges={[
    {kind: 'neutral', icon: '💬', label: 'Command'},
    {kind: 'auth', icon: '🔒', label: 'Auth required'},
  ]}
/>

<ApiCallout kind="warning" icon="🔒" title="Requires">

- Authentication
- Verified account
- Scope: `example.scope`

</ApiCallout>

### Client Message

```json
{
  "route": "@wfm|cmd/example/doThing",
  "id": "message-id",
  "payload": {
    "name": "Example",
    "enabled": true
  }
}
```

### Payload Fields

| Field | Type | Description |
|---|---|---|
| `name` | <ApiBadge kind="neutral" label="string" /> | Required. Display name. Max: `32` characters. |
| `enabled` | <ApiBadge kind="neutral" label="bool" /> | Optional. Whether the example is enabled. Default: `false`. |

### Success Message

```json
{
  "route": "@wfm|cmd/example/doThing:ok",
  "payload": {
    "updated": true
  },
  "id": "message-id"
}
```

### Error Message

```json
{
  "route": "@wfm|cmd/example/doThing:error",
  "payload": {
    "name": "app.field.tooLong"
  },
  "id": "message-id"
}
```

## Event Example

<WsMessage
  type="Event"
  route="@wfm|event/example/changed"
  summary="Notify clients that an example resource changed."
  badges={[{kind: 'neutral', icon: '🔔', label: 'Event'}]}
/>

```json
{
  "route": "@wfm|event/example/changed",
  "payload": {
    "id": "example-id",
    "changedAt": "2026-06-19T00:00:00Z"
  }
}
```

### Payload Fields

| Field | Type | Description |
|---|---|---|
| `id` | <ApiBadge kind="neutral" label="string" /> | Changed resource identifier. |
| `changedAt` | <ApiBadge kind="neutral" label="string" /> | Change timestamp. |

## Stream Example

<WsMessage
  type="Stream"
  route="@wfm|event/example/state"
  summary="Broadcast an ordered state update for one stream."
  badges={[
    {kind: 'neutral', icon: '🔔', label: 'Event'},
    {kind: 'neutral', icon: '🔁', label: 'Stream'},
  ]}
/>

Use stream metadata only when the backend sends `meta.stream` and `meta.revision`.

```json
{
  "route": "@wfm|event/example/state",
  "payload": {
    "state": "ready"
  },
  "meta": {
    "stream": "example:example-id",
    "revision": 7
  }
}
```

### Meta Fields

| Field | Type | Description |
|---|---|---|
| `stream` | <ApiBadge kind="neutral" label="string" /> | Stream name used for ordering or deduplication. |
| `revision` | <ApiBadge kind="neutral" label="number" /> | Monotonic revision within the stream. |

## Model Payload Example

Use `ts` when the message body is schematic and refers to a reusable model.

<WsMessage
  type="Event"
  route="@wfm|event/subscriptions/newOrder"
  summary="Send a matching newly created visible order to subscribers."
  badges={[{kind: 'neutral', icon: '🔔', label: 'Event'}]}
/>

```ts
{
  route: "@wfm|event/subscriptions/newOrder",
  payload: Order,
}
```

Payload is an [Order](../data-models.mdx#order) shape.

## Not Implemented Example

<WsMessage
  type="Command"
  route="@wfm|cmd/example/future"
  summary="Reserved example command that is not implemented yet."
  badges={[{kind: 'danger', icon: '🚧', label: 'Not implemented'}]}
/>

Use this pattern only when a planned or stubbed backend route needs to be mentioned. Do not document stubbed routes as active behavior.

## Copyable Skeleton

Use this skeleton for a new command route.

````mdx
## Message Name

<WsMessage
  type="Command"
  route="@wfm|cmd/domain/action"
  summary="Short behavior description."
  badges={[
    {kind: 'neutral', icon: '💬', label: 'Command'},
  ]}
/>

### Client Message

```json
{
  "route": "@wfm|cmd/domain/action",
  "id": "message-id",
  "payload": {}
}
```

### Payload Fields

| Field | Type | Description |
|---|---|---|
| `field` | <ApiBadge kind="neutral" label="string" /> | Field description. |

### Success Message

```json
{
  "route": "@wfm|cmd/domain/action:ok",
  "id": "message-id"
}
```

### Error Message

```json
{
  "route": "@wfm|cmd/domain/action:error",
  "payload": "app.errors.example",
  "id": "message-id"
}
```
````

## Do Not

- Do not document routes that are constants only; confirm they are registered.
- Do not document stubbed handlers as active behavior.
- Do not document fields that the backend does not implement, such as `refId`.
- Do not render loose badge rows before route text; use `WsMessage`.
- Do not use `json` for model placeholders such as `payload: "Order"`; use `ts` and link the model.
- Do not duplicate reusable model schemas on WebSocket pages.
